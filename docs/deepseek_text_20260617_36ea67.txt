# LLM Firewall – Rust Implementation Project

## 1. Vision & Goals

A **single static binary** that acts as a **transparent Layer‑7 reverse proxy** for any OpenAI‑compatible LLM API.  
It inspects all chat completion requests/responses in real time, enforcing the OWASP Top 10 for LLM Applications, DLP, and token budgets.  
Designed for enterprise security teams: minimal attack surface, predictable ultra‑low latency, audit‑ready logging.

### Key Deliverable
- Binary `llm-firewall`  
- Config file `llm-firewall.yaml`  
- systemd unit & Docker example  
- Works with **existing LiteLLM, raw OpenAI, Azure, vLLM, etc.** without modifying those backends.

## 2. Architecture Overview
[Client] --(HTTPS/TLS optional)--> [llm-firewall:8080] --(HTTP)--> [Upstream: e.g., LiteLLM:4000]

text

**Inside the firewall (Rust async process)**:

- `main.rs`: starts the HTTP server (hyper/actix-web), loads config, initializes pipeline.
- `proxy/handler.rs`: receives client request, orchestrates inspection, forwards to upstream, returns response to client (streaming or non‑streaming).
- `pipeline/engine.rs`: runs an ordered chain of `Detector` trait objects on the request, then on the response.
- `detectors/*.rs`: each detector implements `Detector`.
- `config.rs`: YAML deserialization + environment variable overrides.
- `utils/*.rs`: token counting, audit logging, metrics.

**Inspection flow** (per request):

1. Client POST to `/v1/chat/completions` (and other endpoints later).
2. Firewall parses JSON body (serde).
3. Request pipeline runs:
   - Prompt injection detection
   - DLP scanning (with redact/block action)
   - System prompt enforcement (inject/mandatory)
   - Token budget check
   - Tool call validation
   - Rate limiting
4. If blocked → return HTTP 4xx with structured error.
5. Otherwise, forward modified request to upstream (inject API key, remove client key).
6. For streaming (`stream: true`):
   - Buffer entire response (up to configurable max) before inspection.
   - Run response pipeline (DLP, output sanitization).
   - If violation, truncate stream and inject error message; else stream cleaned response to client.
7. Log audit record (JSON to stdout or Kafka).

| **Metrics** | `metrics` + `metrics-exporter-prometheus` |

5.2 Detector Trait (detectors/mod.rs)
rust
use async_trait::async_trait;
use crate::pipeline::context::{RequestContext, ResponseContext};
use crate::errors::DetectorError;

#[async_trait]
pub trait Detector: Send + Sync {
    /// Display name for logging
    fn name(&self) -> &'static str;

    /// Inspect and potentially modify the request before upstream call.
    /// Return Ok(()) if allowed, or Err(DetectorError::Blocked(reason)) to block.
    async fn inspect_request(&self, ctx: &mut RequestContext) -> Result<(), DetectorError>;

    /// Inspect the full (accumulated) response text.
    async fn inspect_response(&self, ctx: &mut ResponseContext) -> Result<(), DetectorError>;
}
5.3 Pipeline (pipeline/engine.rs)
rust
pub struct Pipeline {
    request_detectors: Vec<Box<dyn Detector>>,
    response_detectors: Vec<Box<dyn Detector>>,
}

impl Pipeline {
    pub async fn run_request(&self, ctx: &mut RequestContext) -> Result<(), DetectorError> {
        for det in &self.request_detectors {
            det.inspect_request(ctx).await?; // short-circuits on error
        }
        Ok(())
    }

    pub async fn run_response(&self, ctx: &mut ResponseContext) -> Result<(), DetectorError> {
        for det in &self.response_detectors {
            det.inspect_response(ctx).await?;
        }
        Ok(())
    }
}
5.4 Request/Response Context (pipeline/context.rs)
rust
pub struct RequestContext {
    pub method: String,
    pub path: String,
    pub headers: HeaderMap,
    pub body: serde_json::Value,          // parsed OpenAI chat request
    pub client_ip: Option<IpAddr>,
    pub api_key: Option<String>,          // extracted from header
    // Detectors can modify body (e.g., redact, inject system prompt)
    pub modified_body: Option<serde_json::Value>,
}

pub struct ResponseContext {
    pub status: u16,
    pub body_text: String,               // accumulated response content
    pub is_stream: bool,
    // Detectors can set an override/error message
    pub override_response: Option<String>,
}
5.5 Proxy Handler (proxy/handler.rs)
Parse incoming HTTP request.

Build RequestContext, run pipeline.run_request().

If blocked → return 403 / 400 with JSON error {"error": {"message": "Blocked by firewall: ..."}}.

Modify request JSON if needed (inject system prompt, redact fields) based on ctx.modified_body.

Forward to upstream using reqwest::Client (with streaming support).

If stream: true in request:

Read upstream stream, accumulate chunks until end of stream or max buffer.

Run response pipeline on accumulated text.

If override: send single chunk with error and data: [DONE].

Else: stream the cleaned text back chunked as SSE.

Log audit.

5.6 Detector Implementations
Injection Detector (injection.rs)

Signature mode (default): load pattern list from file or built‑in; build aho_corasick::AhoCorasick automaton. Scan all messages.content strings (including system and user).

ML mode (feature‑gated): load ONNX model, tokenize input, run inference, check score > threshold.

Action: block or redact (replace matched substrings with [REDACTED]).

DLP Detector (dlp.rs)

Built‑in patterns for SSN, credit card, email, IP, API keys.

Custom patterns from config (regex strings compiled on startup).

Actions per rule: block (immediate rejection), redact (replace in body), log_only.

Applied both to request and response.

Output Sanitizer (output_sanitizer.rs)

Scan for HTML/JS injection attempts in response: <script, javascript:, data:text/html, dangerous markdown/HTML attributes.

Can strip or block.

Tool Call Inspector (tool_call.rs)

Validates that any function/tool calls in the request (and later response) are in the allowed list.

Blocks hallucinated tools in responses.

Token Budget (token_budget.rs)

Count prompt tokens using tokenizers (must select tokenizer based on model; default to cl100k).

Maintain in‑memory map (DashMap) of user/API key token usage per time window.

Block if exceeds per‑request or per‑window limit.

Rate Limiter (rate_limiter.rs)

Use governor with user/API key as key.

Block with HTTP 429 Too Many Requests.

6. Streaming Handling Strategy
The firewall must buffer the entire response text up to max_response_buffer (default 32KB) before running response detectors.
Procedure:

When upstream responds with 200 and Content-Type: text/event-stream, we start forwarding chunks immediately to the client? No – for security we must inspect the complete text.

Actually, we can stream to client and simultaneously buffer; if a violation is found later, we can close the connection abruptly or inject an error. This is complex.

For MVP, use accumulate‑then‑forward: wait for all chunks, run inspection, then send cleaned stream or block. This adds latency equal to response generation time, which is acceptable for an initial release. We'll document this limitation.

7. Logging & Audit
Use tracing with JSON layer (tracing-subscriber). Emit structured logs to stdout (or Kafka with rdkafka in future).
Log every request with:

Correlation ID (generate if not present)

Client IP, API key hash

Detectors results (pass/block, reasons)

Token counts

Masked prompt and response (after DLP redaction)

8. Deployment
Binary
Build with cargo build --release, produces target/release/llm-firewall.
Statically link musl for portability: rustup target add x86_64-unknown-linux-musl.

Systemd Unit (llm-firewall.service)

Description=LLM Firewall

Restart=always

[Install]

11. Key Design Decisions
No external service dependency: All inspection runs in‑process. (Redis can be added later for distributed token budget.)

Configuration via YAML + env vars: fits enterprise config management.

Pluggable detectors: easy to add new OWASP categories or custom checks.

Streaming trade‑off: accumulate full response for inspection – safe, simple, good for compliance. Acceptable latency.

Stateless proxy: no session affinity required, horizontal scaling trivial.

12. Future Enhancements (v1.1+)

Support for Anthropic Messages API, Gemini, etc.
